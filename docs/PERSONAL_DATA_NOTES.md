# PERSONAL_DATA_NOTES

Документ описывает только техническую реализацию согласия и хранения данных. Это не юридическое заключение по 152-ФЗ.

## 1. Что реализовано технически

Перед использованием сценариев бот показывает сообщение о согласии с документами, ссылки на документы и кнопку `✅ Согласен, продолжить`.

До согласия бот не создает заявки и не запускает сценарии:

- `Записаться лично`;
- `Описать ситуацию`;
- `Система лояльности`;
- `Контакты`.

После согласия бот записывает событие в `personal_data_consents`:

- `client_id`;
- `telegram_id`;
- `consent_text_version`;
- `source`;
- `accepted_at`;
- `personal_data_policy_url`;
- `user_agreement_url`;
- `personal_data_consent_url`;
- `public_offer_url`.

Версия текста согласия задается через `CONSENT_TEXT_VERSION`.

## 2. Какие данные собираются

В MVP бот может сохранять:

- Telegram ID;
- username;
- first name;
- last name;
- ссылку на Telegram-профиль;
- текст обращения пользователя;
- тип заявки;
- статус заявки;
- дату согласия;
- версию текста согласия.

Поля `phone` и `email` есть в таблице `clients`, но бот в MVP их не запрашивает.

## 3. Где хранятся ссылки на документы

Ссылки задаются в env:

```env
PERSONAL_DATA_POLICY_URL=https://disk.yandex.ru/...
USER_AGREEMENT_URL=https://disk.yandex.ru/...
PERSONAL_DATA_CONSENT_URL=https://disk.yandex.ru/...
PUBLIC_OFFER_URL=https://disk.yandex.ru/...
```

Если любая ссылка не заполнена, приложение не запускается.

## 4. Что должен проверить юрист

- Текст политики обработки персональных данных.
- Текст согласия внутри бота.
- Сроки хранения данных.
- Порядок удаления данных по запросу пользователя.
- Нужно ли отдельное согласие на рассылки, если они появятся позже.
- Нужны ли дополнительные документы для NocoDB/Directus и сотрудников, работающих с базой.

## 5. Технические рекомендации

- Не писать персональные данные в логи.
- Ограничить доступ к PostgreSQL и NocoDB/Directus.
- Использовать HTTPS в production.
- Включить резервные копии PostgreSQL.
- Хранить production-данные в инфраструктуре РФ.
